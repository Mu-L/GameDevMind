## 为什么需要热更新？

| 维度 | 内容 |
|------|------|
| **作用** | 在不重新发布应用包的情况下，动态更新游戏内容、功能和数据 |
| **应用场景** | 内容和功能的更新；调整策划数据；修复产品bug；快速响应运营需求 |

| 运营需求 | 说明 |
|----------|------|
| **数据表热更** | 配置数据热更新 |
| **资源+代码热更** | 资源与代码动态更新 |

| 问题 | 解决方向 |
|------|----------|
| **如何应对快速变化的需求？** | 使用热更新快速响应需求；避免频繁发版；提升运营效率 |
| **如何避免用户流失？** | 通过热更避免用户流失；游戏用户不用下载安装新包；提升用户体验 |

| 要点和思考方向 |
|----------------|
| 热更新是网游产品快速迭代的基础能力 |
| 可以避免频繁发版导致的用户流失 |
| 需要支持差异化更新，不同渠道、版本可能需要不同的热更内容 |

## 热更新系统需要具备的功能

| 维度 | 内容 |
|------|------|
| **作用** | 提供完整的热更新功能，支持快速制作、发布和管理热更新 |
| **应用场景** | 热更新数据制作；热更新版本发布；热更新版本管理；热更新版本回滚 |

| 核心功能 | 说明 |
|----------|------|
| **差异化更新** | 支持不同渠道、版本的热更内容 |
| **快速制作和发布** | 提升运营效率 |
| **避免用户流失** | 游戏用户不用下载安装新包 |
| **一键生成可热更数据** | 简化数据制作流程 |
| **一键发布数据** | 自动上传、预热 |
| **可快速切换版本** | 支持版本回滚 |
| **版本管理** | 坐标：环境、平台、分支等 |

| 问题 | 解决方向 |
|------|----------|
| **如何实现差异化更新？** | 支持不同渠道、版本的热更内容；实现差异化更新机制 |
| **如何提升制作和发布效率？** | 实现一键生成可热更数据；实现一键发布数据（自动上传和预热） |
| **如何实现版本管理？** | 实现多维度版本管理（环境、平台、分支）；支持快速切换版本和回滚 |

| 要点和思考方向 |
|----------------|
| 热更新系统要易用，降低运营成本 |
| 版本管理要灵活，支持多环境、多平台、多分支 |
| 需要支持快速回滚，应对线上问题 |

## 客户端方案

| 维度 | 内容 |
|------|------|
| **作用** | 在客户端实现热更新功能，支持动态加载资源和代码 |
| **应用场景** | 游戏客户端需要支持热更新；资源动态加载；代码动态更新 |

| 代码热更方案 | 候选 | 说明 |
|--------------|------|------|
| **Lua** | tolua、xlua | 使用脚本语言实现热更新 |
| **C#脚本** | — | 预先插入注入代码 |
| **C#热更** | HybridCLR、ILRuntime | 使用ILRuntime或HybridCLR实现C#热更新 |

| 资源更新功能 | 说明 |
|--------------|------|
| **指向更新后台** | 配置更新源 |
| **检查版本** | 版本检查 |
| **获取资源清单** | 获取资源清单 |
| **比较资源hash值** | 识别变化资源 |
| **同步/下载变化的资源** | 注意：不能直接覆盖原文件，需安全更新机制 |
| **更新时机** | 启动时检查更新；后台静默更新 |

| 依赖/对接系统 | 说明 |
|---------------|------|
| **HTTP网络系统** | 多线程并行下载 |
| **异常处理** | 网络下载异常处理、断点续传、重试机制 |
| **UI系统、元数据系统、脚本系统** | 对接系统 |

| 问题 | 解决方向 |
|------|----------|
| **如何实现代码热更新？** | 使用Lua或ILRuntime/HybridCLR；根据项目需求选择合适的方案 |
| **如何实现资源更新？** | 实现版本检查和资源清单获取；比较资源hash值；安全下载和更新资源；合理选择更新时机 |
| **如何处理网络异常？** | 使用HTTP网络系统；实现断点续传、重试机制 |

| 要点和思考方向 |
|----------------|
| 客户端热更新需要支持资源更新和代码更新 |
| 脚本语言（如Lua）是常见的代码热更新方案 |
| 资源更新要支持差异更新，避免下载完整资源包 |
| 需要考虑网络异常、下载失败等异常情况 |
| 更新时机要合理，避免影响用户体验 |

## 服务端方案

| 维度 | 内容 |
|------|------|
| **作用** | 服务端热更新配置数据，支持动态调整游戏配置 |
| **应用场景** | 服务端需要支持配置数据热更新；游戏配置动态调整；不停服更新配置 |

| 数据特点 | 说明 |
|----------|------|
| **共用配置数据** | 服务端和客户端通常共用配置数据 |
| **不共用艺术和代码** | 服务端通常不需要共用艺术数据和代码 |
| **更新方式** | 数据包整体下载、释放，适合服务端数据特性 |

| 问题 | 解决方向 |
|------|----------|
| **如何实现配置数据热更新？** | 实现配置数据包整体下载和释放；根据服务端数据特性选择合适的更新方式 |
| **如何保证数据一致性？** | 数据更新后同步更新内存中的元数据；更新被使用和引用的数据；更新由元数据衍生出来的数据 |

| 要点和思考方向 |
|----------------|
| 服务端热更新主要是配置数据更新 |
| 需要考虑数据更新后内存数据的同步更新 |
| 服务端数据更新要保证数据一致性 |

## 数据生产方案

| 维度 | 内容 |
|------|------|
| **作用** | 将策划、美术、程序的数据转换为游戏中可使用的格式 |
| **应用场景** | 热更新数据制作流程；数据格式转换；数据打包处理 |

| 数据类型 | 来源 | 处理方式 |

| **配置数据** | 策划 | 生成可使用的配置数据；Excel转json、xml等 |

| **代码数据** | 程序 | 脚本编码加固（可选）；保护知识产权 |

| 问题 | 解决方向 |
|------|----------|
| **如何处理配置数据？** | 生成可使用的配置数据；实现Excel转json、xml等格式转换 |
| **如何处理艺术和场景数据？** | 直接生成bundle；实现资源打包 |
| **如何处理代码数据？** | 支持脚本编码加固；根据需求选择是否加固 |
| **如何实现自动化生产？** | 实现数据生产自动化；支持一键生成可热更数据 |

| 要点和思考方向 |
|----------------|
| 数据生产要自动化，支持一键生成 |
| 不同来源的数据需要不同的处理方式 |
| 代码数据可以考虑加密加固，保护知识产权 |

## 数据发布方案

| 维度 | 内容 |
|------|------|
| **作用** | 将热更新数据发布到CDN，供客户端下载 |
| **应用场景** | 热更新数据发布流程；数据上传和分发；CDN数据管理 |

| 更新方式 | 说明 |
|----------|------|
| **增量包更新** | 过去端游常用 |
| **差异更新** | 只更新变化的部分；减少下载量；提升更新速度 |

| 问题 | 解决方向 |
|------|----------|
| **如何选择数据更新方式？** | 根据项目需求选择；差异更新可大大减少下载量 |
| **如何加速数据下载？** | 使用CDN对数据加速；新上传的数据要预热，更新的文件要刷新 |

| 要点和思考方向 |
|----------------|
| 差异更新可以大大减少下载量，提升更新速度 |
| CDN加速可以提升下载速度，改善用户体验 |
| 需要支持预热和刷新，确保用户能获取到最新数据 |

## 版本管理

| 维度 | 内容 |
|------|------|
| **作用** | 管理热更新版本和分支，支持多维度版本控制 |
| **应用场景** | 热更新版本管理；版本和分支规划；多渠道版本管理 |

| 版本维度 | 说明 |
|----------|------|
| **版本** | 版本维度 |
| **渠道** | 渠道维度 |

| 分支类型 | 说明 |
|----------|------|
| **渠道分支** | 为渠道创建版本控制分支；内容一致的渠道在一个分支上 |
| **热修复分支** | 用于日常快速修复线上问题 |
| **分支合并** | 在热更分支修复，合并回开发分支；保持代码一致性 |

| 问题 | 解决方向 |
|------|----------|
| **如何规划版本和分支？** | 支持多维度版本管理（版本、渠道）；为渠道创建分支；设定热修复分支；实现分支合并 |

| 要点和思考方向 |
|----------------|
| 版本管理要支持多维度（版本、渠道、分支） |
| 热修复分支可以快速修复线上问题，不影响主分支 |
| 分支合并要保持代码一致性 |

## 热更发布流程

| 维度 | 内容 |
|------|------|
| **作用** | 规范热更新发布流程，确保发布质量 |
| **应用场景** | 日常热更新发布；热更新版本发布；热更新质量保证 |

| 流程阶段 | 说明 |
|----------|------|
| **1. 准备** | 准备热更测试环境 |
| **2. 制作** | 制作热更数据 |
| **3. 发布** | 发布热更数据 |
| **4. 测试** | 在测试环境应用版本；用测试包进行测试 |
| **5. 应用** | 在正式服应用热更版本 |

| 问题 | 解决方向 |
|------|----------|
| **如何规范发布流程？** | 建立规范的发布流程；包含准备、制作、发布、测试、应用等阶段 |
| **如何保证发布质量？** | 重视测试环节，避免问题版本上线；支持测试环境和正式环境的切换 |

| 要点和思考方向 |
|----------------|
| 热更发布要有规范的流程，确保质量 |
| 测试环节很重要，避免问题版本上线 |
| 需要支持测试环境和正式环境的切换 |

## 大版本发布

| 维度 | 内容 |
|------|------|
| **作用** | 处理大版本更新时的热更新策略，确保版本兼容性 |
| **应用场景** | 大版本更新；版本兼容性处理；更新策略制定 |

| 兼容情况 | 处理策略 |
|----------|----------|
| **老版本兼容新热更** | 直接制作、发布、应用最新内容的热更 |
| **老版本不兼容新热更** | 只针对最新包开启热更检查；3天内非强制更新提醒；3天后强制更新提醒 |

| 问题 | 解决方向 |
|------|----------|
| **如何处理版本兼容性？** | 评估老版本客户端兼容性；根据兼容情况选择相应策略 |
| **如何制定更新策略？** | 制定合理的更新策略；平衡用户体验和版本统一；谨慎使用强制更新，避免用户流失 |

| 要点和思考方向 |
|----------------|
| 大版本发布要考虑老版本兼容性 |
| 需要制定合理的更新策略，平衡用户体验和版本统一 |
| 强制更新要谨慎，避免用户流失 |
