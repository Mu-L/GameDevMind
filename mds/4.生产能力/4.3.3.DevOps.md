<h2 align="center">DevOps</h2>
<p>
在游戏开发过程中，要不断地出数据和产品的版本来进行测试、调试和发布，这些过程如果全部依赖人工操作是非常低效且容易出错的。
通过建设 DevOps 体系（CI/CD、版本生产线、工程管理等），可以用自动化工具链和规范化流程，大幅提升项目的生产效率和稳定性。
</p>

**关键词:**<br/>
*CI,CD,DevOps,Maven,Jenkins,出包生产线,工程管理*

**标签:**<br/>
*等级: 中级|高级, 阶段: 开发, 分类: 生产能力, 角色: 客户端开发|服务端开发|运维*

## 持续集成（CI）

| 维度 | 内容 |
|------|------|
| **作用** | 最大化地实现生产自动化，降低人力投入，让每次提交都尽快得到「可运行、可测试」的反馈 |
| **应用场景** | 自动化构建、自动化测试、自动化发布、提高开发效率 |

| 问题 | 解决方向 |
|------|----------|
| **如何实现持续集成** 需要实现自动化生产流程 | 设计 CI 流程；实现自动化工具；优化 CI 性能 |

| 应用举例 | 说明 |
|----------|------|
| **游戏包自动构建** | 实时为 QA 提供可测试版本；及时准备可发布版本 |
| **自动测试** | 尽快发现和定位问题；单元测试、性能测试等；每次提交自动运行测试 |
| **数据发布自动化** | 自动提交、发布配置、活动表、资源清单等 |
| **运维辅助** | 批量服务器管理（更新、停服、开服、重启、检查等） |

| 要点和思考方向 |
|----------------|
| 持续集成是提高开发效率的重要手段 |
| 实现自动化减少人力投入 |
| 快速反馈提高问题发现速度 |
| 持续优化 CI 流程提高效率 |

## 持续交付（CD）

| 维度 | 内容 |
|------|------|
| **作用** | 保证代码从提交到交付测试/上线之间的流程尽可能自动化和可重复 |
| **应用场景** | 自动化交付、自动化发布、提高发布效率、降低发布风险 |

| 问题 | 解决方向 |
|------|----------|
| **如何实现持续交付** 需要实现自动化交付流程 | 设计 CD 流程；实现自动化工具；优化 CD 性能 |
| **如何选择辅助工具** 需要选择合适的辅助工具 | 评估工具特性、项目需求；选择合适工具 |

| 典型目标 | 说明 |
|----------|------|
| **持续发布** | 代码提交后可自动/手动触发出包，交付 QA 测试；批量出正式发布版本并归档 |
| **发布自动化** | 尽量减少人工干预步骤，降低出错概率 |

| 辅助工具 | 说明 |
|----------|------|
| **fastlane** | 辅助 iOS 和 Android 应用程序自动化测试版部署和发布 |

| 要点和思考方向 |
|----------------|
| 持续交付是提高发布效率的重要手段 |
| 实现自动化流程减少人工干预 |
| 降低出错概率提高发布质量 |
| 选择合适的工具提高效率 |

## 版本交付生产线

| 维度 | 内容 |
|------|------|
| **作用** | 搭建一条「可以实现批量快速出包」的统一生产线 |
| **应用场景** | 批量出包、多渠道支持、多平台支持、提高出包效率 |

| 问题 | 解决方向 |
|------|----------|
| **如何解决多渠道多平台问题** 渠道多、平台多，对接耗时耗力 | 产品出包：统一出包流程、多渠道支持；对接平台（账户、支付）SDK；实现批量、快速处理 |
| **如何实现生产线流程** 需要实现完整的生产线流程 | 设计生产线流程；实现流程步骤；优化流程性能 |

| 典型生产线流程 | 说明 |
|----------------|------|
| **更新资源、数据** | 资源更新、数据更新 |
| **构建资源** | 资源构建 |
| **生成 bundle** | Bundle 生成 |
| **生成资源清单** | 清单生成 |
| **应用渠道差异信息** | 渠道差异应用 |
| **构建安装包** | 安装包构建 |
| **输出到指定位置** | 本地或制品仓库 |

| 要点和思考方向 |
|----------------|
| 版本交付生产线是批量出包的基础 |
| 解决多渠道多平台问题 |
| 实现批量快速出包 |
| 持续优化生产线流程 |

## 工程管理

| 维度 | 内容 |
|------|------|
| **作用** | 通过合理的工程管理和依赖管理，解决大项目中的分工、协作、复用与依赖问题 |
| **应用场景** | 大型项目管理、模块化管理、依赖管理、团队协作 |

| 问题 | 解决方向 |
|------|----------|
| **如何实现工程管理** 需要解决大项目中的管理问题 | 分工、协作、复用、依赖管理；设计工程管理架构；实现管理机制 |
| **如何实现项目对象模型** 需要实现模块化工程管理 | 将大工程模块化；做好模块部署；管理模块间依赖；设计项目对象模型 |

| 典型关注点 | 说明 |
|-------------|------|
| **分工** | 模块分工、职责划分 |
| **协作** | 团队协作、沟通机制 |
| **复用** | 代码复用、模块复用 |
| **依赖管理** | 依赖解析、依赖管理 |

| 项目对象模型 | 说明 |
|--------------|------|
| **模块化/对象化** | 模块划分、对象化 |
| **坐标** | 版本、分支 |
| **依赖描述与管理** | 依赖描述、依赖管理 |
| **构建与发布流程定义** | 构建流程、发布流程 |
| **候选** | Maven 等 |

| 要点和思考方向 |
|----------------|
| 工程管理是大型项目的基础 |
| 实现模块化管理提高可维护性 |
| 实现依赖管理保证项目稳定 |
| 持续优化工程管理效率 |
