## 6.1 运维技术 — 稳定运行的守护者

### 6.1.1 服务器监控

**监控什么？**

| 指标类型 | 具体指标 | 告警阈值 |
|----------|----------|----------|
| **系统指标** | CPU 使用率 | >80% |
| | 内存使用率 | >85% |
| | 磁盘使用率 | >90% |
| **网络指标** | 带宽使用率 | >80% |
| | 网络延迟 | >100ms |
| | 连接数 | 接近上限 |
| **应用指标** | 请求 QPS | 异常波动 |
| | 错误率 | >1% |
| | 响应时间 | P99 > 1s |
| **业务指标** | 在线人数 | 异常下跌 |
| | 登录成功率 | <95% |
| | 充值金额 | 异常波动 |

**常用监控工具：**

| 工具 | 特点 |
|------|------|
| **Prometheus + Grafana** | 开源、灵活、可视化好 |
| **Zabbix** | 老牌、功能全面 |
| **Datadog** | SaaS、功能强大、价格高 |
| **阿里云监控** | 阿里云用户首选 |

**监控告警原则：**

1. **分级告警**：P0（电话）> P1（短信）> P2（邮件/IM）
2. **告警收敛**：避免告警风暴
3. **告警可操作**：每个告警都要有对应的处理流程
4. **定期回顾**：分析告警有效性，优化阈值

### 6.1.2 故障处理

**故障分级：**

| 级别 | 说明 | 响应时间 | 示例 |
|------|------|----------|------|
| **P0** | 核心功能不可用 | 5 分钟 | 全服宕机、充值失败 |
| **P1** | 重要功能受影响 | 15 分钟 | 部分玩家无法登录 |
| **P2** | 一般功能问题 | 1 小时 | 某个活动入口异常 |
| **P3** | 轻微问题 | 4 小时 | UI 显示问题 |

**故障处理流程：**

```
1. 发现故障（监控告警/用户反馈）
   ↓
2. 初步判断（影响范围、严重程度）
   ↓
3. 应急响应（拉起应急会议、通知相关人员）
   ↓
4. 止损优先（回滚、降级、限流）
   ↓
5. 问题定位（日志分析、排查原因）
   ↓
6. 修复上线（验证后上线修复）
   ↓
7. 复盘总结（写故障报告、制定改进措施）
```

**故障复盘模板：**

```markdown
## 故障概述
- 故障时间：YYYY-MM-DD HH:MM
- 影响范围：XXX
- 故障级别：P0/P1/P2/P3

## 时间线
- HH:MM 故障发生
- HH:MM 发现故障
- HH:MM 开始处理
- HH:MM 故障恢复

## 原因分析
- 直接原因：...
- 根本原因：...

## 改进措施
- [ ] 短期措施（1 周内）
- [ ] 中期措施（1 月内）
- [ ] 长期措施（3 月内）
```

### 6.1.3 容量规划

**容量评估要素：**

| 要素 | 说明 |
|------|------|
| **DAU/MAU** | 日活/月活用户数 |
| **PCU** | 最高同时在线人数 |
| **ARPU** | 每用户平均收入 |
| **服务器规格** | CPU、内存、带宽 |
| **数据库容量** | 数据量、QPS |

**容量规划公式：**

```
服务器数量 = PCU / 单服承载人数 × 冗余系数

冗余系数一般取 1.5-2.0，预留扩容空间
```

**扩容策略：**

| 策略 | 说明 |
|------|------|
| **垂直扩容** | 升级服务器配置（CPU、内存） |
| **水平扩容** | 增加服务器数量 |
| **自动扩容** | 根据负载自动增减服务器 |
| **分区扩容** | 按区域/等级分服 |

## 6.2 产品运营支持 — 让运营更高效

### 6.2.1 GM 后台

**GM 后台核心功能：**

| 功能模块 | 功能说明 |
|----------|----------|
| **用户管理** | 查询用户信息、封禁/解封账号 |
| **物品管理** | 发放/回收物品、修改背包 |
| **数据查询** | 查询用户数据、游戏数据 |
| **邮件系统** | 全服邮件、指定用户邮件 |
| **活动管理** | 配置活动参数、开启/关闭活动 |
| **日志查询** | 操作日志、异常日志 |

**GM 后台设计原则：**

1. **权限控制**：不同角色不同权限
2. **操作审计**：所有操作留痕
3. **二次确认**：敏感操作需要确认
4. **限流保护**：防止误操作影响过大

**权限设计示例：**

```
角色          权限
─────────────────────────────────────
客服         查询用户数据、发送邮件
运营         配置活动、发放奖励
管理员       封禁账号、修改数据
超级管理员   全部权限
```

### 6.2.2 数据分析

**核心数据指标：**

| 指标类型 | 指标 | 说明 |
|----------|------|------|
| **用户规模** | DAU | 日活跃用户 |
| | MAU | 月活跃用户 |
| | 新增用户 | 当日新注册用户 |
| **用户质量** | 留存率 | 次日/7 日/30 日留存 |
| | 在线时长 | 平均在线时长 |
| | 登录频次 | 平均登录次数 |
| **付费数据** | 付费率 | 付费用户/活跃用户 |
| | ARPU | 每用户平均收入 |
| | ARPPU | 每付费用户平均收入 |
| | LTV | 用户生命周期价值 |

**常用分析模型：**

| 模型 | 用途 |
|------|------|
| **漏斗分析** | 分析用户转化路径 |
| **留存分析** | 分析用户留存情况 |
| **分群分析** | 按用户特征分组分析 |
| **行为分析** | 分析用户行为路径 |
| **收入分析** | 分析收入来源和趋势 |

**数据分析工具：**

- **自建**：ELK、ClickHouse、Superset
- **SaaS**：神策数据、GrowingIO、Mixpanel
- **游戏垂直**：ThinkingData、DataEye

## 6.3 技术运营 — 快速迭代的支撑

### 6.3.1 热更新

**为什么需要热更新？**

1. **快速修复**：不用等应用商店审核
2. **小步快跑**：小版本快速迭代
3. **A/B 测试**：快速验证新功能
4. **运营活动**：灵活配置活动

**热更新方案对比：**

| 方案 | 平台 | 优点 | 缺点 |
|------|------|------|------|
| **Lua** | 全平台 | 成熟、生态好 | 需要集成 Lua 虚拟机 |
| **ILRuntime** | Unity | 纯 C#、性能好 | 只支持 Unity |
| **HybridCLR** | Unity | 原生支持、性能好 | 较新、需要验证 |
| **小程序** | 微信 | 原生支持 | 仅限小程序平台 |

**热更新流程：**

```
1. 开发更新内容
   ↓
2. 打包资源/代码
   ↓
3. 上传到 CDN
   ↓
4. 更新版本配置文件
   ↓
5. 客户端检测更新
   ↓
6. 下载并应用更新
   ↓
7. 重启/热重载
```

**热更新注意事项：**

1. **版本兼容**：新旧版本数据兼容
2. **回滚机制**：更新失败能回滚
3. **增量更新**：只下载变化的部分
4. **强制更新**：重大更新强制用户更新

---

### 6.3.2 A/B 测试

**什么是 A/B 测试？**

将用户分成不同组，每组看到不同的版本，通过数据对比哪个版本更好。

**适用场景：**

- UI 设计：哪个按钮颜色点击率高
- 玩法调整：哪个难度设置留存好
- 付费点：哪个价格点收入高
- 新手引导：哪个流程转化好

**A/B 测试流程：**

```
1. 提出假设（"红色按钮比蓝色按钮点击率高"）
   ↓
2. 设计实验（A 组红色，B 组蓝色）
   ↓
3. 确定样本量（需要多少用户才有统计意义）
   ↓
4. 分流（随机将用户分到 A/B 组）
   ↓
5. 运行实验（收集数据）
   ↓
6. 分析结果（统计显著性检验）
   ↓
7. 决策（采用优胜版本或继续优化）
```

**注意事项：**

- **单一变量**：每次只测试一个变量
- **样本充足**：确保统计显著性
- **时间足够**：避免短期波动影响
- **伦理考虑**：不要欺骗用户

## 6.4 商业化支持 — 让游戏可持续发展

### 6.4.1 付费设计

**常见付费模式：**

| 模式 | 说明 | 代表游戏 |
|------|------|----------|
| **买断制** | 一次性付费 | 单机游戏、主机游戏 |
| **F2P** | 免费 + 内购 | 大多数手游 |
| **订阅制** | 按月/年付费 | Xbox Game Pass |
| **Battle Pass** | 赛季通行证 | 王者荣耀、Fortnite |
| **广告变现** | 看广告得奖励 | 休闲小游戏 |

**付费点设计原则：**

1. **不破坏平衡**：付费不能严重影响公平性
2. **物有所值**：让玩家觉得值得
3. **循序渐进**：从低到高，引导付费
4. **多元选择**：提供不同价位的商品

**常见付费点：**

| 类型 | 示例 |
|------|------|
| **外观类** | 皮肤、时装、头像框 |
| **功能类** | 月卡、特权、加速 |
| **资源类** | 金币、钻石、材料 |
| **抽卡类** | 角色/武器/装备抽取 |
| **成长类** | 经验加成、等级礼包 |

---

### 6.4.2 运营活动

**活动类型：**

| 类型 | 目的 | 示例 |
|------|------|------|
| **拉新活动** | 吸引新用户 | 邀请好友得奖励 |
| **活跃活动** | 提升日活 | 每日签到、在线奖励 |
| **付费活动** | 提升收入 | 充值返利、限时礼包 |
| **留存活动** | 提升留存 | 连续登录奖励 |
| **节日活动** | 借势营销 | 春节活动、周年庆 |

**活动配置系统设计：**

```
活动配置
├── 活动时间（开始/结束）
├── 活动规则（参与条件、完成条件）
├── 活动奖励（奖励内容、发放方式）
├── 活动 UI（界面配置、素材）
└── 数据埋点（效果追踪）
```

**活动效果评估：**

| 指标 | 说明 |
|------|------|
| **参与率** | 参与活动的用户比例 |
| **完成率** | 完成活动的用户比例 |
| **ROI** | 活动投入产出比 |
| **对核心指标影响** | 对 DAU、收入的影响 |

## 🎯 运营能力建设建议

### 对于小团队

- **优先级 1**：基础监控、日志系统
- **优先级 2**：简单的 GM 后台
- **可以暂缓**：复杂的 A/B 测试系统

### 对于中大型团队

- **优先级 1**：完善的监控告警体系
- **优先级 2**：数据分析平台
- **优先级 3**：热更新、A/B 测试能力
