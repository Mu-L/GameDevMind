## 3.1 客户端研发 — 让游戏"动"起来

### 3.1.1 游戏循环

**是什么？**

游戏循环是游戏的核心，它不断重复执行以下操作：

```
while (游戏运行中) {
    1. 处理输入 (Handle Input)
    2. 更新逻辑 (Update)
    3. 渲染画面 (Render)
}
```

**Unity 中的游戏循环：**

```csharp
void Update() {
    // 每帧调用，处理游戏逻辑
    // 受帧率影响
}

void FixedUpdate() {
    // 固定时间间隔调用，处理物理
    // 默认 0.02 秒（50 次/秒）
}

void LateUpdate() {
    // 在 Update 之后调用
    // 常用于摄像机跟随
}

void OnRenderObject() {
    // 渲染时调用
}
```

**常见问题：**

| 问题 | 解决方向 |
|------|----------|
| 帧率波动大 | 使用 FixedUpdate 处理物理，Update 处理逻辑 |
| 逻辑与帧率耦合 | 使用 deltaTime，实现帧率无关的逻辑 |
| 卡顿 | 优化 Update 中的代码，避免频繁分配 |

### 3.1.2 输入处理

**输入类型：**

- **键盘**：WASD 移动、快捷键
- **鼠标**：点击、拖拽、滚轮
- **触摸**：单点、多点、手势
- **手柄**：摇杆、按键、震动
- **体感**：加速度计、陀螺仪

**输入处理最佳实践：**

1. **抽象输入层**：将输入与具体操作解耦
2. **支持多平台**：同一套逻辑适配不同输入设备
3. **输入缓冲**：提升操作手感（如格斗游戏的输入缓存）
4. **输入重映射**：允许玩家自定义按键

### 3.1.3 场景管理

**是什么？**

场景管理负责加载、卸载、切换游戏场景。

**常见方案：**

| 方案 | 说明 | 适用场景 |
|------|------|----------|
| **单场景** | 所有东西在一个场景 | 小游戏、简单项目 |
| **多场景** | 按区域/功能分场景 | 大多数游戏 |
| **流式加载** | 动态加载/卸载区域 | 开放世界 |
| **场景分块** | 将大场景分成小块 | 大型地图 |

**Unity 场景管理示例：**

```csharp
// 异步加载场景
AsyncOperation asyncLoad = SceneManager.LoadSceneAsync("NextScene");

// 设置加载进度
while (!asyncLoad.isDone) {
    float progress = Mathf.Clamp01(asyncLoad.progress / 0.9f);
    // 更新进度条 UI
}

// 多场景同时加载
SceneManager.LoadScene("Scene1", LoadSceneMode.Additive);
SceneManager.LoadScene("Scene2", LoadSceneMode.Additive);
```

### 3.1.4 资源管理

**核心概念：**

| 概念 | 说明 |
|------|------|
| **资源加载** | 从磁盘/网络加载资源到内存 |
| **资源卸载** | 释放不再使用的资源 |
| **资源缓存** | 缓存常用资源，避免重复加载 |
| **对象池** | 复用频繁创建销毁的对象 |
| **资源打包** | 将资源打包成 AssetBundle 等格式 |

**对象池示例：**

```csharp
public class ObjectPool<T> where T : MonoBehaviour {
    private Queue<T> pool = new Queue<T>();
    private T prefab;
    
    public T Get() {
        if (pool.Count > 0) {
            return pool.Dequeue();
        }
        return Instantiate(prefab);
    }
    
    public void Return(T obj) {
        obj.gameObject.SetActive(false);
        pool.Enqueue(obj);
    }
}

// 使用：管理子弹
ObjectPool<Bullet> bulletPool = new ObjectPool<Bullet>(bulletPrefab);
Bullet bullet = bulletPool.Get();
// ... 使用子弹
bulletPool.Return(bullet); // 归还到池中
```

## 3.2 服务端研发 — 稳定可靠的后台支撑

### 3.2.1 架构设计

**常见服务器架构：**

| 架构 | 说明 | 适用场景 |
|------|------|----------|
| **单服** | 所有功能在一个进程 | 小游戏、测试服 |
| **分区服** | 按玩家等级/区域分服 | 传统 MMO |
| **微服务** | 按功能拆分服务 | 大型游戏、全球化 |
| **无缝服** | 多服协同，玩家无感知 | 开放世界 MMO |

**典型微服务拆分：**

```
┌─────────────────────────────────────────┐
│              网关服务 (Gateway)          │
└─────────────────────────────────────────┘
           │         │         │
           ▼         ▼         ▼
    ┌──────────┐ ┌──────────┐ ┌──────────┐
    │ 登录服务 │ │ 战斗服务 │ │ 聊天服务 │
    └──────────┘ └──────────┘ └──────────┘
           │         │         │
           ▼         ▼         ▼
    ┌──────────┐ ┌──────────┐ ┌──────────┐
    │ 用户服务 │ │ 物品服务 │ │ 匹配服务 │
    └──────────┘ └──────────┘ └──────────┘
```

### 3.2.2 数据同步

**同步方案对比：**

| 方案 | 优点 | 缺点 | 适用场景 |
|------|------|------|----------|
| **状态同步** | 安全性高、防作弊 | 带宽占用大、延迟敏感 | MMO、SLG、卡牌 |
| **帧同步** | 带宽小、流畅 | 安全性低、需确定性 | MOBA、RTS、格斗 |

**状态同步示例：**

```csharp
// 客户端发送操作
void SendMoveInput(Vector3 position, Quaternion rotation) {
    network.Send(new MoveMessage {
        playerId = localId,
        position = position,
        rotation = rotation,
        timestamp = Time.time
    });
}

// 服务器广播状态
void BroadcastPlayerState() {
    foreach (var player in players) {
        var state = new PlayerStateMessage {
            playerId = player.id,
            position = player.position,
            hp = player.hp,
            state = player.state
        };
        network.BroadcastToAll(state);
    }
}
```

## 3.3 业务层功能 — 游戏的核心玩法

### 3.3.1 角色系统

**核心模块：**

```
角色系统
├── 基础属性（HP、MP、攻击力、防御力）
├── 等级系统（经验、升级、属性成长）
├── 装备系统（穿戴、强化、镶嵌）
├── 技能系统（学习、升级、释放）
└── 背包系统（物品、整理、使用）
```

**设计要点：**

1. **数据结构设计**：使用配置表驱动，便于策划调整
2. **扩展性**：预留扩展点，方便后续添加新功能
3. **数据验证**：服务端验证所有关键数据，防止作弊

---

### 3.3.2 战斗系统

**战斗类型：**

| 类型 | 特点 | 代表游戏 |
|------|------|----------|
| **回合制** | 轮流行动、策略性强 | 梦幻西游、阴阳师 |
| **即时制** | 实时操作、反应要求高 | 王者荣耀、原神 |
| **半即时** | 结合两者特点 | 一些 MMORPG |

**战斗流程框架：**

```
1. 战斗开始
   ↓
2. 初始化（加载角色、设置初始状态）
   ↓
3. 战斗循环
   ├── 玩家输入/AI 决策
   ├── 技能/动作执行
   ├── 伤害计算
   ├── 状态更新
   └── 胜负判断
   ↓
4. 战斗结束（结算、奖励发放）
```

---

### 3.3.3 社交系统

**核心功能：**

- **好友系统**：添加、删除、黑名单
- **聊天系统**：私聊、群聊、世界频道
- **公会/帮派**：创建、加入、管理
- **组队系统**：组队、匹配、副本
- **交易系统**：交易行、拍卖行、面对面交易

## 🎯 研发流程建议

### 标准研发流程

```
1. 需求分析
   ↓
2. 技术方案设计
   ↓
3. 原型开发（验证可行性）
   ↓
4. 功能开发
   ↓
5. 单元测试
   ↓
6. 集成测试
   ↓
7. 性能优化
   ↓
8. 上线发布
```

### 敏捷开发实践

- **迭代开发**：2 周一个迭代，每个迭代交付可用功能
- **每日站会**：同步进度、解决问题
- **代码审查**：保证代码质量、知识共享
- **持续集成**：自动化测试、快速发现问题
