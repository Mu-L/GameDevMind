## 5.1 工作流管理 — 让团队有序运转

### 5.1.1 敏捷开发在游戏开发中的应用

**敏捷的核心价值观：**

> - **个体和互动** 高于 流程和工具
> - **可工作的软件** 高于 详尽的文档
> - **客户合作** 高于 合同谈判
> - **响应变化** 高于 遵循计划

**Scrum 框架：**

```
┌─────────────────────────────────────────┐
│            Sprint (2-4 周)               │
│                                          │
│  Sprint 计划 → 每日站会 → Sprint 评审 → 回顾  │
│       ↑                              │    │
│       └────────── 产品待办 ──────────┘    │
└─────────────────────────────────────────┘
```

**关键角色：**

| 角色 | 职责 |
|------|------|
| **Product Owner** | 定义产品愿景、管理需求优先级 |
| **Scrum Master** | 确保流程顺利、移除障碍 |
| **开发团队** | 自组织、跨职能、交付产品 |

**游戏开发中的敏捷实践：**

| 实践 | 说明 |
|------|------|
| **Sprint 计划** | 每 2 周规划一次迭代内容 |
| **每日站会** | 15 分钟同步进度和问题 |
| **看板** | 可视化任务状态（To Do/Doing/Done） |
| **回顾会议** | 每个 Sprint 结束后复盘改进 |

---

### 5.1.2 任务分配

**任务分解原则：**

1. **粒度适中**：每个任务 4-16 小时能完成
2. **责任明确**：每个任务有且只有一个负责人
3. **依赖清晰**：明确任务之间的依赖关系
4. **可验收**：有明确的完成标准（DoD）

**任务分配技巧：**

| 技巧 | 说明 |
|------|------|
| **能力匹配** | 根据成员能力分配任务 |
| **成长空间** | 给成员适当挑战，促进成长 |
| **工作平衡** | 避免有人过载、有人闲置 |
| **主动认领** | 鼓励成员主动认领任务，提升参与感 |

**任务管理工具：**

- **Jira**：功能强大，适合大团队
- **Trello**：轻量级，适合小团队
- **Asana**：界面友好，功能全面
- **飞书/钉钉**：国内团队常用

---

### 5.1.3 进度跟踪

**关键指标：**

| 指标 | 说明 |
|------|------|
| **燃尽图** | 显示剩余工作量随时间的变化 |
| **velocity** | 每个 Sprint 完成的故事点数 |
| **任务完成率** | 按时完成的任务比例 |
| **阻塞任务数** | 当前被阻塞的任务数量 |

**进度预警信号：**

- ⚠️ 连续 3 个 Sprint velocity 下降
- ⚠️ 任务完成率低于 70%
- ⚠️ 阻塞任务超过 5 个
- ⚠️ 关键路径任务延期

## 5.2 质量保障 — 质量是做出来的，不是测出来的

### 5.2.1 代码审查（Code Review）

**为什么要做 Code Review？**

1. **发现 Bug**：第二双眼睛更容易发现问题
2. **知识共享**：团队成员了解彼此代码
3. **统一风格**：保持代码风格一致
4. **提升质量**：促进编写更好的代码

**Code Review 检查清单：**

```
□ 功能正确性：代码是否实现了预期功能
□ 代码规范：命名、格式是否符合规范
□ 性能考虑：有无明显的性能问题
□ 错误处理：是否妥善处理了异常情况
□ 测试覆盖：是否有相应的单元测试
□ 文档注释：关键逻辑是否有注释
□ 代码复用：有无重复代码可以提取
```

**Code Review 最佳实践：**

| 实践 | 说明 |
|------|------|
| **小批量** | 每次 Review 不超过 400 行代码 |
| **及时响应**：24 小时内回复 Review 意见 |
| **对事不对人**：评论代码，不评论人 |
| **建设性意见**：指出问题的同时给出建议 |
| **自动化优先**：用工具检查格式、规范 |

**Code Review 工具：**

- **GitHub Pull Request**：最常用
- **GitLab Merge Request**：GitLab 用户
- **Phabricator**：Facebook 开源
- **Gerrit**：Google 开源

---

### 5.2.2 测试策略

**测试金字塔：**

```
        ╱╲
       ╱  ╲      E2E 测试（少量）
      ╱────╲
     ╱      ╲    集成测试（适量）
    ╱────────╲
   ╱          ╲  单元测试（大量）
  ╱────────────╲
```

**各层测试说明：**

| 层级 | 测试对象 | 比例 | 工具 |
|------|----------|------|------|
| **单元测试** | 单个函数/类 | 70% | NUnit、xUnit |
| **集成测试** | 模块间交互 | 20% | Unity Test Framework |
| **E2E 测试** | 完整流程 | 10% | 自动化 UI 测试 |

**游戏测试的特殊性：**

1. **视觉测试**：画面效果需要人工检查
2. **手感测试**：操作手感需要主观评估
3. **平衡性测试**：数值平衡需要反复调整
4. **兼容性测试**：多设备、多平台测试

---

### 5.2.3 质量标准

**代码质量标准：**

| 指标 | 目标值 |
|------|--------|
| 单元测试覆盖率 | >80% |
| 代码重复率 | <5% |
| 圈复杂度 | <15 |
| 技术债比率 | <5% |

**Definition of Done (DoD) 示例：**

```
□ 代码已完成并通过自测
□ 代码已提交并通过 CI
□ 单元测试已编写并全部通过
□ Code Review 已完成
□ 相关文档已更新
□ 策划/美术已验收（如适用）
```

## 5.3 项目管理 — 在约束中找平衡

### 5.3.1 进度控制

**影响进度的因素：**

| 因素 | 应对策略 |
|------|----------|
| **需求变更** | 控制变更频率，评估影响 |
| **技术风险** | 提前验证，准备备选方案 |
| **人员变动** | 知识共享，降低 Bus Factor |
| **外部依赖** | 提前沟通，预留缓冲 |

**进度控制技巧：**

1. **关键路径法**：识别并重点关注关键路径任务
2. **缓冲管理**：在关键节点预留缓冲时间
3. **里程碑管理**：设置明确的里程碑，定期检查
4. **透明沟通**：及时同步进度和风险

---

### 5.3.2 风险管理

**风险识别：**

| 风险类型 | 示例 |
|----------|------|
| **技术风险** | 新技术不成熟、性能达不到要求 |
| **人员风险** | 核心人员离职、关键岗位空缺 |
| **进度风险** | 任务延期、依赖延期 |
| **质量风险** | Bug 过多、性能问题 |
| **外部风险** | 政策变化、平台规则变化 |

**风险管理流程：**

```
风险识别 → 风险评估 → 风险应对 → 风险监控
```

**风险应对策略：**

| 策略 | 说明 | 示例 |
|------|------|------|
| **规避** | 改变计划，避开风险 | 不用不成熟的新技术 |
| **转移** | 将风险转移给第三方 | 外包给专业团队 |
| **减轻** | 降低风险发生概率或影响 | 增加测试、代码审查 |
| **接受** | 接受风险，准备应急方案 | 预留缓冲时间 |

## 5.4 团队管理 — 人是最大的变量

### 5.4.1 沟通协作

**有效沟通的原则：**

1. **明确目标**：沟通前想清楚要达成什么
2. **倾听理解**：先理解对方，再寻求被理解
3. **及时反馈**：收到信息后及时回应
4. **选择合适的渠道**：紧急事打电话，复杂事当面聊

**常见沟通场景：**

| 场景 | 推荐方式 |
|------|----------|
| **需求讨论** | 面对面会议 + 文档记录 |
| **进度同步** | 每日站会 + 看板 |
| **问题讨论** | 小范围会议 + 即时通讯 |
| **决策通知** | 邮件/文档 + 会议确认 |
| **反馈辅导** | 一对一当面沟通 |

---

### 5.4.2 人才培养

**技术人员成长路径：**

```
初级 → 中级 → 高级 → 专家 → 架构师
  │      │      │      │       │
  └──────┴──────┴──────┴───────┘
              │
          持续学习
```

**培养方法：**

| 方法 | 说明 |
|------|------|
| **导师制** | 新人配导师，快速融入 |
| **技术分享** | 定期分享，促进学习 |
| **轮岗** | 接触不同模块，拓宽视野 |
| **外部培训** | 参加行业会议、培训课程 |
| **挑战性任务** | 给适当挑战，促进成长 |

**反馈技巧（SBI 模型）：**

- **Situation**：描述具体情境
- **Behavior**：描述具体行为
- **Impact**：说明行为的影响

**示例：**
> "在昨天的代码审查中（Situation），你提交的代码没有单元测试（Behavior），这导致我们无法确认功能是否正确，增加了上线风险（Impact）。"

### 5.4.3 团队建设

**高效团队的特征：**

1. **目标一致**：团队成员理解并认同目标
2. **信任互助**：成员之间互相信任、互相帮助
3. **沟通顺畅**：信息透明，沟通无障碍
4. **责任共担**：共同承担责任，不推诿
5. **持续改进**：不断反思，持续进步

**团队建设活动：**

| 类型 | 活动示例 |
|------|----------|
| **技术类** | Hackathon、技术分享、代码道场 |
| **社交类** | 团队聚餐、桌游、户外活动 |
| **学习类** | 读书会、在线课程小组 |
| **庆祝类** | 里程碑庆祝、生日会 |

## 🎯 管理能力建设建议

### 对于新晋管理者

1. **角色转变**：从"做事"到"带人"
2. **学会授权**：不要事必躬亲
3. **建立信任**：言出必行，公平公正
4. **持续学习**：管理是一门需要学习的技能

### 对于资深管理者

1. **文化建设**：塑造积极的团队文化
2. **人才培养**：培养下一代管理者
3. **战略思考**：从更高视角思考问题
4. **影响力建设**：在组织内建立影响力
